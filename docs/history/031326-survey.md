# rsac — Cross-Platform Audio Capture: Technical Survey and Gap Analysis

**Date:** 2025-03-13
**Status:** Living reference document
**Scope:** Full audit of capture level support, streaming architecture, identified gaps, and closure roadmap

---

## 1. Executive Summary

rsac is a streaming-first, pull-model Rust audio capture library for Windows (WASAPI), Linux (PipeWire), and macOS (CoreAudio). Its core abstraction is a lock-free SPSC ring buffer bridge that decouples OS real-time audio callbacks from consumer threads. Phases 0-4 are architecturally complete. Phase 5 (breadth expansion) is in progress.

**Primary differentiated value vs cpal:** per-application and per-process-tree audio capture — capabilities cpal does not expose at all. Device and system level capture work on all platforms but are also covered by cpal. rsac's unique capability is the application capture layer.

**Primary user requirement:** rsac as a streaming source for downstream consumers within a program — piping captured audio to DSP, encoding, analysis pipelines, or other components — not just file writing.

**Audit result:** 10 gaps identified:
- 3 bugs — wrong behavior being produced or advertised
- 4 feature gaps — correct API path missing or stubbed
- 3 streaming enhancements — missing async/fan-out/observability

## 2. Architecture Overview

### 2.1 Core Data Flow

OS real-time thread pushes via BridgeProducer::push_or_drop() into an rtrb SPSC ring buffer.
Consumer thread pulls via BridgeStream::read_chunk() (blocking) or try_read_chunk() (non-blocking).

Builder chain: AudioCaptureBuilder -> .build() -> AudioCapture -> .start() -> ``BridgeStream<S>`` (implements CapturingStream)

WASAPI event loop       -> f32 samples -> BridgeProducer::push_or_drop()
PipeWire process cb     -> f32 samples -> BridgeProducer::push_or_drop()
CoreAudio AUHAL IOProc  -> f32 samples -> BridgeProducer::push_or_drop()
                                               |
                                    [rtrb SPSC ring buffer]
                                               |
                                    BridgeConsumer -> BridgeStream
                                               |
                              read_chunk() / try_read_chunk() -> AudioBuffer

### 2.2 Module DAG (strict, no reverse dependencies)

core/ -> bridge/ -> audio/ -> api/ -> lib.rs

### 2.3 CaptureTarget Enum (src/core/config.rs)

| Variant | ID Type | Meaning |
|---|---|---|
| SystemDefault | — | OS default output loopback or monitor |
| Device(DeviceId) | String | Specific audio device by platform ID |
| Application(ApplicationId) | String (PID or PW node serial) | Specific app session |
| ApplicationByName(String) | String | First running app matching name (case-insensitive) |
| ProcessTree(ProcessId) | u32 | Parent process and all children by PID |

### 2.4 PlatformStream Trait (src/bridge/stream.rs)

Internal backend contract: stop_capture() and is_active(). `BridgeStream<S>` wraps any PlatformStream and provides the full CapturingStream implementation. All three backends use BridgeStream — no per-platform duplication of the consumer pattern.

### 2.5 Sink Adapters (src/sink/)

| Sink | Purpose |
|---|---|
| NullSink | Discard all data (testing/benchmarking) |
| ChannelSink | std::sync::mpsc — caller gets Receiver<AudioBuffer> for downstream |
| WavFileSink | File writing (behind sink-wav feature) |

## 4. Platform Deep Dives

### 4.1 Windows (WASAPI) — src/audio/windows/thread.rs

#### What Works

create_audio_client() maps all five CaptureTarget variants:

- SystemDefault: DeviceEnumerator::new() -> get_default_device(Render) -> get_iaudioclient() -> standard loopback
- Device: find_device_by_id() enumerates DeviceCollection, matches ID string. Falls back to default if ID is "default" or empty.
- Application(pid): app_id.0.parse::<u32>() -> AudioClient::new_application_loopback_client(pid, false) — correct and working.
- ProcessTree(pid): AudioClient::new_application_loopback_client(pid.0, true) with include_tree=true maps to PROCESS_LOOPBACK_MODE_INCLUDE_TARGET_PROCESS_TREE — fully implemented.

#### Windows Process Loopback API Mechanics (from wasapi-rs)

Uses ActivateAudioInterfaceAsync with AUDIOCLIENT_ACTIVATION_TYPE_PROCESS_LOOPBACK:
1. Build AUDIOCLIENT_ACTIVATION_PARAMS with TargetProcessId and ProcessLoopbackMode
2. Wrap in PROPVARIANT as VT_BLOB
3. ActivateAudioInterfaceAsync(VIRTUAL_AUDIO_DEVICE_PROCESS_LOOPBACK, IAudioClient IID, params, handler)
4. Block on Condvar until ActivateCompleted fires
5. GetActivateResult -> extract IAudioClient
6. initialize_client with ShareMode::Shared, autoconvert=true

Critical constraints:
- Shared mode ONLY — exclusive not supported for process loopback
- autoconvert flag MANDATORY — engine handles format mismatch
- For ProcessTree: MUST supply the parent PID, not a child PID
- Non-functional methods in this mode: get_mixformat, is_supported, get_device_period, get_buffer_size, get_current_padding, get_audiorenderclient, get_audiosessioncontrol, get_audioclock

#### Windows Gaps

[G1] ApplicationByName silently falls back to SystemDefault — Priority P2
- File: src/audio/windows/thread.rs — CaptureTarget::ApplicationByName arm
- Current: Logs a warning, returns default render device IAudioClient. User gets system audio, not named app audio. Silent to the caller.
- Fix: sysinfo is already a Cargo dependency. Use System::new_all().processes_by_name(OsStr::new(name)), get parent PID, call new_application_loopback_client(pid, false).
- Reference: wasapi-rs/examples/record_application.rs shows this exact pattern.

[G2] PlatformCapabilities bug: process_tree_capture = false — Priority P1
- File: src/core/capabilities.rs — windows() function
- Current: supports_process_tree_capture: false
- Reality: ProcessTree is fully implemented via include_tree=true
- Fix: Change to true — one line.
- Impact: Users gate feature availability on this flag. Currently blocked unnecessarily.

### 4.2 Linux (PipeWire) — src/audio/linux/thread.rs

#### What Works

Stream properties set before stream.connect():
- Base (all targets): NODE_NAME="rsac-capture", STREAM_CAPTURE_SINK="true", STREAM_MONITOR="true"
- SystemDefault: No TARGET_OBJECT — STREAM_CAPTURE_SINK + STREAM_MONITOR route to default sink monitor
- Device: TARGET_OBJECT = device_id.0 (PipeWire node serial) — correct
- Application: TARGET_OBJECT = app_id.0 — correct when caller provides a valid node serial

#### Two PipeWire Capture Strategies (from obs-pipewire-audio-capture + wiremix)

Strategy A — TARGET_OBJECT (single node):
Set TARGET_OBJECT to the node's object.serial. PipeWire routes that node's output to your capture stream. Requires prior node discovery. Used for Application and Device.

Strategy B — Virtual Sink + Dynamic Links (OBS plugin pattern, multi-node):
1. Create virtual null-audio-sink via pw_core_create_object with factory="support.null-audio-sink", media.class="Stream/Input/Audio", node.virtual="true"
2. Attach registry listener watching for PW_TYPE_INTERFACE_Node globals with media.class="Stream/Output/Audio"
3. For each matching node, create link-factory objects: PW_KEY_LINK_OUTPUT_NODE/PORT -> virtual sink input ports
4. Connect capture stream targeting the virtual sink's serial as TARGET_OBJECT
5. As new matching nodes appear in the registry, add new links dynamically

#### Node Discovery Properties

| Property | PipeWire Key | Use |
|---|---|---|
| application.name | PW_KEY_APP_NAME | For name matching |
| application.process.binary | PW_KEY_APP_PROCESS_BINARY | Executable name matching |
| object.serial | PW_KEY_OBJECT_SERIAL | What to use as TARGET_OBJECT |
| media.class | PW_KEY_MEDIA_CLASS | Filter: "Stream/Output/Audio" |
| client.id | PW_KEY_CLIENT_ID | Bind to Client object for more metadata |

#### Linux Gaps

[G3] ApplicationByName passes name raw as TARGET_OBJECT — Priority P2
- File: src/audio/linux/thread.rs — CaptureTarget::ApplicationByName arm
- Current: props.insert(TARGET_OBJECT, name.as_str()) — WRONG. TARGET_OBJECT expects object.serial or node.name, not application.name.
- Fix: Before stream creation, run a registry scan on the PipeWire thread. Subscribe to registry globals, collect PW_TYPE_INTERFACE_Node globals with media.class="Stream/Output/Audio", read application.name and application.process.binary, match case-insensitively, use matched node's object.serial as TARGET_OBJECT.
- New file: src/audio/linux/registry.rs for node enumeration helper.

[G4] ProcessTree is single-process only — Priority P2
- File: src/audio/linux/thread.rs — CaptureTarget::ProcessTree arm
- Current: Passes pid.0.to_string() as TARGET_OBJECT. PipeWire does not map PIDs as node serials — category error.
- Fix: Implement Strategy B. Create virtual null-audio-sink, enumerate all Stream/Output/Audio nodes, match by application.process.binary for target executable, link all matching node output ports to the virtual sink, watch registry for new matches and link dynamically.
- New file: src/audio/linux/virtual_sink.rs
- Complexity: High — PipeWire object lifecycle management and dynamic link creation.

### 4.3 macOS (CoreAudio) — src/audio/macos/thread.rs + tap.rs

#### What Works

resolve_capture_target() maps all five variants:
- SystemDefault: get_default_device_id(false) -> AUHAL on default output device — correct
- Device: DeviceId.0.parse::<u32>() -> AudioDeviceID -> AUHAL — correct
- Application(pid): app_id.0.parse::<u32>() -> pid -> CoreAudioProcessTap::new(pid, name) -> tap.id() as AUHAL device
- ApplicationByName: enumerate_audio_applications() via NSWorkspace, case-insensitive substring match on name, PID -> tap — fully implemented and correct

#### CoreAudio Process Tap API Mechanics (from AudioCap + audio-rec, macOS 14.4+ minimum)

Step 1 — PID to AudioObjectID:
kAudioHardwarePropertyTranslatePIDToProcessObject on system object with PID qualifier

Step 2 — Create CATapDescription:
initStereoMixdownOfProcesses: [processObjectID]
Configure: uuid=UUID(), muteBehavior, privateTap=true, exclusive=false

Step 3 — Create Tap:
AudioHardwareCreateProcessTap(tapDescription, &tapID)

Step 4 — Create Aggregate Device (REQUIRED — CURRENTLY MISSING IN rsac):
- Read default output UID: kAudioHardwarePropertyDefaultSystemOutputDevice -> AudioDeviceID -> kAudioDevicePropertyDeviceUID -> outputUID string
- Build aggregate device dict:
  kAudioAggregateDeviceNameKey: "Tap-{pid}"
  kAudioAggregateDeviceUIDKey: fresh UUID string
  kAudioAggregateDeviceIsPrivateKey: true
  kAudioAggregateDeviceIsStackedKey: false
  kAudioAggregateDeviceTapAutoStartKey: true
  kAudioAggregateDeviceSubDeviceListKey: [{kAudioSubDeviceUIDKey: outputUID}]
  kAudioAggregateDeviceTapListKey: [{kAudioSubTapDriftCompensationKey: true, kAudioSubTapUIDKey: tapDesc.uuid}]
- AudioHardwareCreateAggregateDevice(dict, &aggregateDeviceID)

Step 5 — Set up AUHAL on aggregateDeviceID (NOT tapID)
Step 6 — AudioDeviceCreateIOProcIDWithBlock on aggregateDeviceID
Step 7 — AudioDeviceStart(aggregateDeviceID, deviceProcID)

Cleanup (must reverse): AudioDeviceStop -> AudioDeviceDestroyIOProcID -> AudioHardwareDestroyAggregateDevice -> AudioHardwareDestroyProcessTap

Required permissions:
- macOS 14.4+ (Sonoma) minimum
- NSAudioCaptureUsageDescription in Info.plist
- com.apple.security.device.audio-input entitlement for sandboxed apps
- TCC: kTCCServiceAudioCapture

#### macOS Gaps

[G5] CRITICAL: Missing aggregate device in ProcessTap — Priority P1
- File: src/audio/macos/tap.rs — CoreAudioProcessTap struct and new() function
- Current: Creates CATapDescription -> AudioHardwareCreateProcessTap -> stores tap_id -> returns tap_id from .id(). Never creates aggregate device.
- Current usage in thread.rs: Passes tap.id() as AudioDeviceID to kAudioOutputUnitProperty_CurrentDevice on AUHAL — WRONG per both reference implementations.
- Impact: Both AudioCap (Swift) and audio-rec (C++) always create an aggregate device and use aggregateDeviceID for AUHAL, not tapID. Without this, AUHAL likely fails to initialize or produces silence.
- Fix: Extend CoreAudioProcessTap to store aggregate_device_id: AudioObjectID. After AudioHardwareCreateProcessTap, read default output UID, build aggregate device dict with tap UUID in TapListKey, call AudioHardwareCreateAggregateDevice, store result. Return aggregate_device_id from .id(). In Drop: destroy aggregate before destroying tap.

[G6] ProcessTree creates single-PID tap — Priority P2
- File: src/audio/macos/thread.rs — CaptureTarget::ProcessTree arm
- Current: CoreAudioProcessTap::new(pid.0, name) — identical to Application arm. Comment says "same as single Application capture for now."
- Fix: Add CoreAudioProcessTap::new_tree(pid) constructor. Enumerate child PIDs (via sysinfo or proc_listpids/proc_pidinfo). Translate each to AudioObjectID via kAudioHardwarePropertyTranslatePIDToProcessObject. Call CATapDescription with the full array: initStereoMixdownOfProcesses: [parentObjID, child1ObjID, ...].

[G7] PlatformCapabilities claims process_tree_capture = true prematurely — Priority P1
- File: src/core/capabilities.rs — macos() function
- Current: supports_process_tree_capture: true
- Reality: Single-PID stub only. Even Application capture is potentially broken due to G5.
- Fix: Change to false until G5 and G6 are both complete.

## 5. Streaming Output Design

### 5.1 Requirement

rsac should function as a streaming audio source for downstream consumers within a program. A caller should be able to pipe captured audio to DSP processing, encoding, analysis, transcription, network forwarding, or other program components — not just files.

### 5.2 Current Streaming Capabilities

| Feature | Status |
|---|---|
| Blocking pull: read_chunk() | Working — blocks until AudioBuffer available |
| Non-blocking poll: try_read_chunk() | Working — returns None if empty |
| mpsc channel sink: ChannelSink | Working — caller gets Receiver<AudioBuffer> |
| WAV file sink: WavFileSink | Working — behind sink-wav feature |
| Async futures_core::Stream | Not implemented — feature flag + AtomicWaker wired, impl missing |
| Fan-out to multiple consumers | Not implemented — SPSC ring buffer design |
| Dropped-frame notification | Not exposed — buffers_dropped counter exists internally |
| AudioCapture::subscribe() convenience | Not implemented |

### 5.3 ChannelSink Pattern (Works Today)

The primary mechanism for downstream streaming within a program is ChannelSink:

    use rsac::{AudioCaptureBuilder, CaptureTarget};
    use rsac::sink::{ChannelSink, AudioSink};

    let mut capture = AudioCaptureBuilder::new()
        .with_target(CaptureTarget::ApplicationByName("spotify".to_string()))
        .sample_rate(48000)
        .channels(2)
        .build()?;

    let (mut sink, receiver) = ChannelSink::new();
    capture.start()?;

    std::thread::spawn(move || {
        while let Ok(buffer) = receiver.recv() {
            let samples: &[f32] = buffer.data();
            let frames: usize = buffer.num_frames();
            // pass to DSP, encoder, analyzer, etc.
        }
    });

### 5.4 Missing: Async Stream Interface

For async downstream consumers (tokio, async-std), futures_core::Stream is not yet implemented on BridgeStream.

Implementation path (G8):
- BridgeShared already has AtomicWaker (behind async-stream feature)
- push() already calls self.shared.waker.wake() after a successful push
- CapturingStream already has register_waker() and is_stream_producing() (behind async-stream feature)
- Missing: impl futures_core::Stream<Item = AudioBuffer> for `BridgeStream<S>` with poll_next that calls try_read_chunk(), registers waker on None, returns Poll::Pending

### 5.5 Streaming Gap Details

[G8] No async Stream implementation — Priority P2
- Files: src/bridge/stream.rs, new src/bridge/async_stream.rs
- Feature flag: async-stream (already in Cargo.toml)
- Approach: poll_next -> try_read_chunk() -> Some: Poll::Ready, None: register_waker() + Poll::Pending. Wake already fires in producer push().

[G9] No AudioCapture::subscribe() convenience — Priority P3
- File: src/api.rs
- Add: pub fn subscribe(&self) -> std::sync::mpsc::Receiver<AudioBuffer> that internally creates and attaches a ChannelSink.

[G10] No dropped-frame/overrun counter exposed — Priority P3
- File: src/api.rs
- BridgeShared.buffers_dropped: AtomicU64 is already incremented by push_or_drop() and exposed via BridgeProducer::buffers_dropped().
- Missing: Not surfaced on AudioCapture. Add AudioCapture::overrun_count() -> u64.

## 9. rsac vs cpal

cpal scope: Device-level capture only (AudioSink/AudioSource nodes), push-model callbacks, no application-level capture, no CaptureTarget concept, BackendSpecificError is an opaque string. Works well for microphone or system loopback at device granularity.

rsac scope: Application-level capture is the primary value — Windows Process Loopback, PipeWire node targeting, CoreAudio Process Tap. Pull-model via BridgeStream. Typed CaptureTarget with 5 variants. 21-variant AudioError with ErrorKind and Recoverability. PlatformCapabilities for honest feature reporting.

They are complementary, not competing. rsac fills the gap cpal intentionally does not cover.

### 11.5 Phase E — Streaming Enhancements (G8, G9, G10)

**G8: Async Stream implementation**
- Unit test: Create a BridgeStream in test mode (using MockPlatformStream, same pattern as existing bridge tests). Push 5 AudioBuffers via BridgeProducer. Use futures::executor::block_on with stream.next() via futures_util::StreamExt. Assert all 5 buffers are received in order.
- Waker test: Create a stream with an empty ring buffer. Register a waker. Push a buffer from another thread 100ms later. Assert the waker fires and poll_next returns Poll::Ready.

### 11.7 CI Integration Notes

- Linux: PipeWire session must be running. Use pipewire --version to verify. Docker-based testing with a virtual PW session is available in docker/.
- macOS: Requires macOS 14.4+. TCC permissions for audio capture must be granted to the test runner. Cannot be tested in sandboxed CI without entitlements. Use a dedicated macOS runner with pre-granted permissions.
- Windows: Process loopback requires Windows 10 build 20348+. CI runner must be Windows Server 2022 or Windows 11. The test audio-producing process must be launched before the capture test starts.
- All platforms: Tests that require real audio hardware should be gated behind an integration feature flag (e.g., #[cfg(feature = "integration-tests")]) to avoid failures in pure unit test environments.</User>
