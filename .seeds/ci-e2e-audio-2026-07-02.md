# CI E2E Audio Seeds - 2026-07-02

Purpose: define end-to-end coverage for rsac and capture runner-dependent work that should not be merged as speculative CI gates without platform evidence.

## E2E Definition

An rsac end-to-end gate means: build the library, CLI/examples, and publishable bindings; provision a deterministic virtual audio route; play a generated 440 Hz tone; capture through the public API/CLI or binding surface; assert non-silence, RMS, 440 Hz tone dominance, negotiated format, and terminal lifecycle behavior after stop/request_stop/drop.

Target capture modes:

- `SystemDefault`
- `Device(DeviceId)` against the virtual monitor/input endpoint
- `Application(ApplicationId)` using the spawned tone player PID
- `ApplicationByName` using the exact runner-observed process/application name
- `ProcessTree(ProcessId)` rooted at the tone player parent

## Seed: Linux Blacksmith Deterministic PipeWire E2E

Current status: `.github/workflows/ci-audio-tests.yml` creates a PipeWire/Pulse null sink and monitor on `blacksmith-4vcpu-ubuntu-2404`, but the workflow intentionally does not set `RSAC_CI_AUDIO_DETERMINISTIC=1` because dbus-less Firecracker PipeWire routing has previously exposed `ci_test_sink.monitor` while `SystemDefault` still yielded zero buffers.

Do not harden Linux content assertions until this evidence exists:

- `pw-cli info 0` ready after manual `pipewire`, `wireplumber`, and `pipewire-pulse` startup.
- `pactl list short sinks` shows only or default-routed `ci_test_sink`.
- `pactl list short sources` shows `ci_test_sink.monitor`.
- A spawned player (`pw-play`, `paplay`, or `sox play`) is explicitly routed to `ci_test_sink` using `PULSE_SINK=ci_test_sink` or a verified PipeWire node link.
- `cargo test --test ci_audio system_capture --no-default-features --features feat_linux -- --test-threads=1 --nocapture` captures non-silent buffers and detects the 440 Hz tone with `RSAC_CI_AUDIO_DETERMINISTIC=1`.

Workflow sketch once proven:

```yaml
- name: Route test tone to PipeWire null sink
  run: |
    PULSE_SINK=ci_test_sink pw-play /tmp/test_440hz.wav &
    pw-dump --no-colors > pipewire-before-test.json
    echo "RSAC_CI_AUDIO_DETERMINISTIC=1" >> "$GITHUB_ENV"
```

Acceptance criteria:

- Linux system, device, application PID, application name, process tree, subscribe, overrun, and lifecycle tests hard-fail on missing tone only after the deterministic route is proven green.
- Preserve soft assertions when `RSAC_CI_AUDIO_DETERMINISTIC` is unset.

## Seed: Windows GitHub-Hosted Full VB-CABLE E2E

Current status: `windows-system` on `windows-latest` installs LABSN/VB-CABLE, verifies VB-CABLE as default playback, and sets `RSAC_CI_AUDIO_DETERMINISTIC=1`. `windows-device` and `windows-process` hard-gate build/start/panic behavior but keep content assertions soft.

Do not set deterministic mode for device/process tiers until this evidence exists:

- The test tone player is still alive after warmup and routes to VB-CABLE playback.
- Device capture targets the corresponding VB-CABLE capture endpoint, not an unrelated default endpoint.
- Process loopback captures the spawned tone process with non-silence and 440 Hz tone dominance for at least two consecutive runs.
- `ApplicationByName` resolves the runner-observed player process name exactly and captures that process.

Workflow sketch once proven:

```yaml
- name: Verify VB-CABLE capture endpoint receives tone
  shell: pwsh
  run: |
    Import-Module AudioDeviceCmdlets -ErrorAction Stop
    Get-AudioDevice -List | Format-Table Index, Type, Name, Default -AutoSize
    "RSAC_CI_AUDIO_DETERMINISTIC=1" >> $env:GITHUB_ENV
```

Acceptance criteria:

- Keep Blacksmith Windows compile/unit-only; its Server image lacks `AudioSrv`, `AudioEndpointBuilder`, and audio endpoints.
- Promote Windows device/process content assertions only on `windows-latest` or a self-hosted Windows runner with an audio stack.

## Seed: macOS Self-Hosted TCC-Gated E2E

Current status: hosted macOS jobs are advisory or skip early because Process Tap system/application/process-tree capture requires `kTCCServiceAudioCapture`, which cannot be granted non-interactively on Blacksmith/GitHub-hosted macOS.

Self-hosted runner requirements:

- macOS 14.4+ with a logged-in user session.
- BlackHole 2ch or equivalent virtual route installed and selected for the deterministic test path.
- The runner service host binary pre-granted Screen & System Audio Recording / Audio Capture permission.
- `RSAC_CI_MACOS_TCC_GRANTED=1` set only after a preflight confirms Process Tap can be created without prompting or hanging.

Workflow sketch for a self-hosted runner label:

```yaml
macos-self-hosted-e2e:
  runs-on: [self-hosted, macos, rsac-audio]
  env:
    RSAC_CI_AUDIO_AVAILABLE: "1"
    RSAC_CI_AUDIO_DETERMINISTIC: "1"
    RSAC_CI_MACOS_TCC_GRANTED: "1"
  steps:
    - uses: actions/checkout@v5
    - uses: dtolnay/rust-toolchain@1.95.0
    - run: cargo test --test ci_audio --no-default-features --features feat_macos -- --test-threads=1 --nocapture
```

Acceptance criteria:

- Managed macOS runners remain compile/unit/advisory only.
- Full macOS capture E2E runs only on a pre-consented self-hosted runner or real hardware.
